## Project Overview

A Rust parser for JMS SQL-like selector expressions, generated from a CongoCC grammar (`SqlExprParser.ccc`). The parser handles boolean/comparison/arithmetic expressions with SQL keywords (AND, OR, NOT, BETWEEN, LIKE, IN, IS NULL, etc.).

## Build & Run Commands

```bash
cargo build              # Build the project
cargo run                # Run the binary (src/main.rs)
cargo test               # Run all tests (summary only)
cargo test -- --nocapture  # Run tests with pretty-printed AST output visible
cargo build --features serde  # Build with serde serialization support
```

## Architecture

The parser is a hand-written recursive-descent parser generated from a CongoCC grammar, using an arena-based AST.

### Key Components

- **`SqlExprParser.ccc`** - CongoCC grammar definition (source of truth for the language). Originally derived from an Apache ActiveMQ JMS selector parser.
- **`src/tokens.rs`** - Token types (`TokenType` enum). Numbered tokens (Token17-Token30) map to operator symbols: `=` `<>` `>` `>=` `<` `<=` `(` `,` `)` `+` `-` `*` `/` `%`.
- **`src/lexer.rs`** - Lexer with case-insensitive keyword matching. Skips whitespace; produces tokens with offset tracking.
- **`src/parser.rs`** - Recursive-descent parser. Entry point: `Parser::new(input).parse()` returns `ParseResult<NodeId>`.
- **`src/arena.rs`** - Arena allocator owning all AST nodes and tokens. `NodeId`/`TokenId` are type-safe indices (newtype over `usize`). Contains the `AstNode` enum (12 variants) and per-node structs. Also has `pretty_print()` with pass-through node elision.
- **`src/visitor.rs`** - Depth-first AST visitor via `Arena::visit()` with `VisitControl` (Continue/SkipChildren/Stop).
- **`src/error.rs`** - `ParseError` with position/line/column tracking.

### AST Expression Hierarchy (precedence low to high)

`JmsSelector` -> `OrExpression` -> `AndExpression` -> `EqualityExpression` -> `ComparisonExpression` -> `AddExpression` -> `MultExpr` -> `UnaryExpr` -> `PrimaryExpr` -> `Literal` | `Variable` | `StringLitteral`

### Usage Pattern

```rust
let mut parser = Parser::new(input.to_string())?;
let root_id = parser.parse()?;
let arena = parser.arena();
// Access nodes: arena.get_node(node_id)
// Pretty print: arena.pretty_print(root_id, 0, parser.input())
// Traverse: arena.visit(root_id, &mut callback, None)
```

### Important Notes

- Source files in `src/` are marked "Generated by CongoCC Parser Generator. Do not edit." but they are the working code and can be modified directly (the CongoCC generator targets Java, not Rust—the Rust code was derived/adapted from the grammar).
- `AddExpressionNode` and `MultExprNode` store `operators: Vec<AddOp/MultExprOp>` between children, where `operators[i]` sits between `children[i]` and `children[i+1]`.
- `EqualityExpressionNode` stores `operators: Vec<EqualityOp>` (Equal, NotEqual, IsNull, IsNotNull). IS NULL/IS NOT NULL are postfix-unary — they push an operator but no right-hand child.
- `ComparisonExpressionNode` stores `operators: Vec<ComparisonOp>` (GreaterThan, GreaterThanEqual, LessThan, LessThanEqual, Like, NotLike, Between, NotBetween, In, NotIn).
- `UnaryExprNode` stores `operator: Option<UnaryOp>` (Plus, Negate, Not) for prefix unary operators.
- `pretty_print()` displays operators in `[...]` after the node name (e.g., `ComparisonExpression [>]`, `UnaryExpr [NOT]`). Nodes with operators are never elided as pass-through.
- The `tests/main_test` file references a different parser crate (`arithmetic`) and is not part of this project's test suite.
- **`tests/pretty_parse_test.rs`** - 10 integration tests validating pretty-print output. Each test uses two logical operators and at least two comparison operators. Covers all expression types and all operator variants.
