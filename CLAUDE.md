## Project Overview

A Rust parser and evaluator for JMS SQL-like selector expressions, generated from a CongoCC grammar (`SqlExprParser.ccc`). The parser handles boolean/comparison/arithmetic expressions with SQL keywords (AND, OR, NOT, BETWEEN, LIKE, IN, IS NULL, etc.).

## Build & Run Commands

```bash
cargo build              # Build the project
cargo run                # Run the binary (src/main.rs)
cargo test               # Run all tests (summary only)
cargo test -- --nocapture  # Run tests with pretty-printed AST output visible
cargo build --features serde  # Build with serde serialization support
```

## Architecture

The parser is a hand-written recursive-descent parser generated from a CongoCC grammar, using an arena-based AST.

### Key Components

- **`SqlExprParser.ccc`** - CongoCC grammar definition (source of truth for the language). Originally derived from an Apache ActiveMQ JMS selector parser.
- **`src/tokens.rs`** - Token types (`TokenType` enum). All variants have descriptive UPPER_SNAKE_CASE names: operators (`EQ`, `NE`, `GT`, `GE`, `LT`, `LE`, `PLUS`, `MINUS`, `STAR`, `SLASH`, `PERCENT`), delimiters (`LPAREN`, `COMMA`, `RPAREN`), whitespace (`SPACE`, `TAB`, `NEWLINE`, `CR`, `FORM_FEED`), comments (`LINE_COMMENT`, `BLOCK_COMMENT`), keywords, and literals (`DECIMAL_LITERAL`, `HEX_LITERAL`, `OCTAL_LITERAL`, `FLOATING_POINT_LITERAL`, `STRING_LITERAL`).
- **`src/lexer.rs`** - Lexer with case-insensitive keyword matching. Skips whitespace, line comments (`--`), and block comments (`/* */`). Produces tokens with offset tracking. Supports scientific notation (`1e5`, `3.14e-2`), long suffix (`42L`), dollar-sign identifiers (`$var`), and escaped single quotes (`'It''s'`).
- **`src/parser.rs`** - Recursive-descent parser. Entry point: `Parser::new(input).parse()` returns `ParseResult<NodeId>`. Enforces boolean root expressions (rejects standalone literals, arithmetic, and strings). Parse-time type checking for BETWEEN bounds and IN list elements.
- **`src/arena.rs`** - Arena allocator owning all AST nodes and tokens. `NodeId`/`TokenId` are type-safe indices (newtype over `usize`). Contains the `AstNode` enum (12 variants) and per-node structs. Also has `pretty_print()` with pass-through node elision.
- **`src/evaluator.rs`** - Expression evaluator. Entry point: `evaluate(input, &variables)` returns `Result<bool, EvalError>`. Supports `RuntimeValue` types (Integer, Float, String, Boolean, Null) for variable binding. Handles all operators including LIKE pattern matching with ESCAPE, BETWEEN, IN, IS NULL/IS NOT NULL. Numeric type promotion (int to float) for mixed arithmetic/comparison.
- **`src/visitor.rs`** - Depth-first AST visitor via `Arena::visit()` with `VisitControl` (Continue/SkipChildren/Stop).
- **`src/error.rs`** - `ParseError` with position/line/column tracking.

### AST Expression Hierarchy (precedence low to high)

`JmsSelector` -> `OrExpression` -> `AndExpression` -> `EqualityExpression` -> `ComparisonExpression` -> `AddExpression` -> `MultExpr` -> `UnaryExpr` -> `PrimaryExpr` -> `Literal` | `Variable` | `StringLiteral`

### Usage Pattern

```rust
let mut parser = Parser::new(input.to_string())?;
let root_id = parser.parse()?;
let arena = parser.arena();
// Access nodes: arena.get_node(node_id)
// Pretty print: arena.pretty_print(root_id, 0, parser.input())
// Traverse: arena.visit(root_id, &mut callback, None)
```

### Important Notes

- Source files in `src/` are marked "Generated by CongoCC Parser Generator. Do not edit." but they are the working code and can be modified directly (the CongoCC generator with prototype support was used to generate the Rust parser from the grammar).
- `AddExpressionNode` stores `operators: Vec<AddOp>` (Plus, Minus) and `MultExprNode` stores `operators: Vec<MultExprOp>` (Star, Slash, Percent) between children, where `operators[i]` sits between `children[i]` and `children[i+1]`.
- `EqualityExpressionNode` stores `operators: Vec<EqualityOp>` (Equal, NotEqual, IsNull, IsNotNull). IS NULL/IS NOT NULL are postfix-unary — they push an operator but no right-hand child.
- `ComparisonExpressionNode` stores `operators: Vec<ComparisonOp>` (GreaterThan, GreaterThanEqual, LessThan, LessThanEqual, Like, NotLike, Between, NotBetween, In, NotIn).
- `UnaryExprNode` stores `operator: Option<UnaryOp>` (Plus, Negate, Not) for prefix unary operators.
- `pretty_print()` displays operators in `[...]` after the node name (e.g., `ComparisonExpression [>]`, `UnaryExpr [NOT]`). Nodes with operators are never elided as pass-through.
- The `tests/main_test` file references a different parser crate (`arithmetic`) and is not part of this project's test suite.

### Lexer Details

- **Comments**: `--` starts a line comment (to end of line/EOF); `/* ... */` is a block comment (error if unterminated). Both are skipped in `skip_ignored()`.
- **Numeric literals**: Integers (`DECIMAL_LITERAL`), hex (`0x`/`0X` prefix → `HEX_LITERAL`), octal (leading `0` + octal digits → `OCTAL_LITERAL`). All integer types accept an optional `L`/`l` long suffix. Numbers with a decimal point or exponent (`e`/`E` with optional `+`/`-`) are emitted as `FLOATING_POINT_LITERAL`.
- **String literals**: Single-quoted. Escaped quotes use `''` (two consecutive single quotes inside a string).
- **Identifiers**: Start with ASCII letter, `_`, or `$`; continue with alphanumeric, `_`, or `$`. Keywords are matched case-insensitively.
- **`--` is a line comment**, not double-negate. Use `- (-x)` for double negation.

### Parser Details

- **Boolean root enforcement**: `parse_jms_selector()` validates that the parsed expression is boolean via `validate_boolean_root()`. Rejects standalone numeric literals, string literals, arithmetic expressions, and NULL. Accepts: comparisons, logical operators (AND/OR/NOT), boolean literals (TRUE/FALSE), and standalone variables (assumed boolean in JMS).
- **BETWEEN bounds**: Only literal values (numeric or string, optionally signed with `+`/`-`). Validates same type category (both numeric or both string) and lower <= upper at parse time.
- **IN list elements**: Only literal values (string, integer, or float, optionally signed with `+`/`-`). All elements must be the same type (Integer, Float, or String — no mixing Integer/Float). `FLOATING_POINT_LITERAL` maps to Float; `DECIMAL_LITERAL`/`HEX_LITERAL`/`OCTAL_LITERAL` map to Integer.

- **`tests/parser_test.rs`** — 164 parser tests covering AST structure for all expression types, operator precedence, error cases, hex/octal/long literals in expressions, and boolean root enforcement.
- **`tests/parser_test2.rs`** — 155 comprehensive parser tests covering all language features including comments, escaped quotes, dollar-sign identifiers, scientific notation, long literals, negative numbers in IN lists, and rejection of non-boolean top-level expressions. - **`tests/parser_type_checking_tests.rs`** — 97 tests for parse-time type checking of BETWEEN bounds and IN list elements, including positive/negative cases, signed bounds, mixed formats, error message quality.  This test was ported from the [sqlexpr-rust](https://github.com/richcar58/sqlexpr-rust) project to validate parity between the two parsers.
- **`tests/pretty_parse_test.rs`** — 10 integration tests validating pretty-print output. Each test uses two logical operators and at least two comparison operators. Covers all expression types and all operator variants.  
- **`tests/evaluator_test.rs`** — 130 evaluator tests covering all operators, type coercion, error cases, NULL handling, LIKE pattern matching, BETWEEN, IN, and edge cases.
- **`tests/evaluator_test2.rs`** — 191 evaluator tests with variable bindings covering arithmetic, comparisons, LIKE with ESCAPE, BETWEEN, IN, IS NULL/IS NOT NULL, boolean logic, and complex expressions.  This test was ported from the [sqlexpr-rust](https://github.com/richcar58/sqlexpr-rust) project to validate parity between the two evaluators.
